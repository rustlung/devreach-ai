# Roadmap проекта DevReach AI

Проект: backend-сервис лендинга разработчика с REST API, обработкой обращения, AI-анализом, email-уведомлениями, логированием, тестами и ручным деплоем на Render.

Python: **3.12**. Локальный запуск: **только `.venv`**. Docker, Docker Compose и `render.yaml` не используются.

## 1. Фундамент проекта

**Цель:** создать минимальный проверяемый FastAPI-каркас.

**Задачи:** структура проекта, настройки через окружение, SQLite/SQLAlchemy, Alembic, логирование, CORS, глобальные ошибки, `GET /api/health`, стартовые тесты, CLI-проверка, правила, roadmap, test-plan, AI-log.

**Проверяемый результат:** `alembic upgrade head`, `pytest tests -vv` и `python -m app.cli check-foundation` проходят локально без внешних API.

**Внешние сервисы:** не используются.

**Критерий завершения:** health проверяет БД, логи пишутся в консоль и файл, тесты этапа зелёные, документация создана.

Статус: ✅ Завершён

## 2. Схемы, нормализация и валидация

**Цель:** подготовить входные и выходные схемы обращения.

**Задачи:** Pydantic-схемы, нормализация однострочных полей, сохранение форматирования многострочного комментария, понятные ошибки валидации, CLI-проверка схем, unit-тесты.

**Проверяемый результат:** данные нормализуются и валидируются без API, БД, OpenAI и email.

**Внешние сервисы:** не используются.

**Критерий завершения:** все правила нормализации покрыты тестами и отражены в test-plan.

Статус: ✅ Завершён

## 3. Модель БД и репозиторий

**Цель:** реализовать сохранение обращений в SQLite.

**Задачи:** SQLAlchemy-модель обращения, Alembic-миграция, `ContactRepository`, методы создания/получения/обновления, логирование операций БД, изолированные тесты.

**Проверяемый результат:** обращение можно создать, получить и обновить без OpenAI и email.

**Внешние сервисы:** не используются.

**Критерий завершения:** миграция применима, репозиторий покрыт unit/integration тестами.

Статус: ✅ Завершён

## 4. AI-сервис

**Цель:** добавить AI-анализ обращения с безопасным fallback.

**Задачи:** OpenAI-клиент, структурированный результат, системный промпт, валидация ответа, обработка отсутствия ключа/таймаута/API-ошибки/некорректного ответа, fake AI service, fixtures, CLI fake/live.

**Проверяемый результат:** AI-сервис проверяется fake-режимом без токенов.

**Внешние сервисы:** ProxyAPI как OpenAI-совместимый шлюз только в отдельной live-команде с явным флагом.

**Критерий завершения:** fallback протестирован, live-вызов по умолчанию невозможен.

Статус: ✅ Автоматическая, fake- и ручная live-проверка через ProxyAPI завершены.

## 5. Email-сервис

**Цель:** реализовать email-уведомления владельцу и пользователю.

**Задачи:** Resend-клиент, HTML/text шаблоны, Jinja2 escaping, fake email service, статусы отправки, логирование, CLI render/fake/live, тесты ошибок провайдера и локальная проверка sender-конфигурации.

**Проверяемый результат:** шаблоны и отправка проверяются без реальных писем.

**Внешние сервисы:** Resend только в отдельной live-команде с явным флагом.

**Критерий завершения:** реальные письма не отправляются случайно, статусы сохраняются.

Статус: ✅ Автоматическая, render- и fake-проверка завершены. Live-проверка Resend ожидает ручного запуска.

## 6. Основной pipeline и `POST /api/contact`

**Цель:** соединить валидацию, БД, AI и email в единый сценарий.

**Задачи:** `ContactService`, dependency injection, endpoint, сохранение обращения до AI/email, fallback, обновление статусов, связка логов через `request_id` и `contact_id`, integration-тесты.

**Проверяемый результат:** pipeline работает с fake AI и fake email.

**Внешние сервисы:** не используются по умолчанию.

**Критерий завершения:** обращение не теряется при сбое AI/email, API возвращает понятный результат.

Статус: ✅ Unit, integration, CLI fake pipeline и regression-проверка завершены локально.

## 7. Rate limiting и защита формы

**Цель:** защитить форму от простого спама.

**Задачи:** rate limiting для `POST /api/contact`, переменные окружения лимитов, HTTP 429, логирование превышений, honeypot, тесты.

**Проверяемый результат:** лишние запросы блокируются до pipeline.

**Внешние сервисы:** не используются.

**Критерий завершения:** превышение лимита не создаёт запись в БД.

## 8. Метрики и расширенный health check

**Цель:** добавить диагностические endpoint.

**Задачи:** расширить health, добавить `GET /api/metrics`, обезличенная статистика по обращениям, AI fallback и email-статусам, тесты.

**Проверяемый результат:** состояние сервиса и агрегаты доступны без персональных данных.

**Внешние сервисы:** не используются.

**Критерий завершения:** метрики не раскрывают содержимое обращений.

## 9. Jinja2-лендинг

**Цель:** создать минимальный интерфейс для демонстрации API.

**Задачи:** главная страница, форма обращения, JavaScript-отправка, состояния загрузки/успеха/ошибок, ручные сценарии.

**Проверяемый результат:** основной сценарий проверяется в браузере.

**Внешние сервисы:** только через уже подготовленный backend pipeline и его режимы.

**Критерий завершения:** frontend не дублирует backend-валидацию как единственный источник истины.

## 10. Полное тестирование

**Цель:** проверить систему перед деплоем.

**Задачи:** завершить unit/integration/e2e тесты, проверить fallback, ошибки email, rate limiting, обработчики ошибок, логи, один контролируемый live OpenAI и одно тестовое письмо.

**Проверяемый результат:** автоматические тесты зелёные, ручной checklist заполнен.

**Внешние сервисы:** OpenAI и email только контролируемо и явно.

**Критерий завершения:** все обязательные проверки отмечены в `docs/test-plan.md`.

## 11. Деплой на Render

**Цель:** развернуть рабочий сервис вручную на Render Web Service.

**Задачи:** подключить GitHub, указать Python 3.12, build/start commands, env vars, миграции при старте, проверить health/Swagger/form/OpenAI/email/logs.

**Проверяемый результат:** получена рабочая публичная ссылка.

**Внешние сервисы:** Render, OpenAI и email в рамках ручной проверки.

**Критерий завершения:** деплой проверен, ограничения SQLite и файловых логов на Render описаны.

## 12. Финальная документация и сдача

**Цель:** подготовить проект к проверке работодателем.

**Задачи:** README, запуск, переменные окружения, стек, архитектура, API, curl-примеры, AI/fallback, хранение данных, логирование, rate limiting, деплой, использование Codex, промпты, ручные исправления, проверка секретов.

**Проверяемый результат:** репозиторий можно поднять с чистого клонирования.

**Внешние сервисы:** только финальные контролируемые проверки.

**Критерий завершения:** документация полная, checklist закрыт, проект готов к отправке.
