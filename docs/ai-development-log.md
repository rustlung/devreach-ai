## Логирование

Настрой подробное и понятное логирование.

Логи должны одновременно выводиться:

- в консоль;
- в файл.

Используй ротацию файлов, например `RotatingFileHandler`.

Формат должен включать минимум:

- дату и время;
- уровень;
- имя logger;
- понятное событие или сообщение;
- request ID, если он доступен;
- путь и метод HTTP-запроса;
- HTTP-статус;
- длительность запроса.

Добавь middleware запросов.

Для каждого HTTP-запроса логируй:

1. начало обработки;
2. завершение;
3. метод;
4. путь;
5. статус;
6. длительность;
7. request ID.

При необработанной ошибке логируй:

- этап;
- тип исключения;
- текст исключения;
- traceback;
- request ID.

Не используй абстрактные сообщения вроде:

```text
Something went wrong
```

Сообщение должно объяснять, где и что произошло.

Не логируй:

- пароли;
- токены;
- секреты;
- полное содержимое пользовательских обращений;
- лишние персональные данные.

В ключевых и неочевидных местах добавляй русские комментарии, объясняющие причину решения. Не комментируй очевидные строки.

## Цель этапа

Создать Pydantic-схемы входных и выходных данных обращения, а также реализовать понятную нормализацию и строгую валидацию:

- имени;
- телефона;
- email;
- комментария;
- служебного honeypot-поля, если оно уже предусмотрено архитектурой или может быть добавлено без реализации endpoint.

Вся логика этапа должна проверяться изолированно:

- без FastAPI endpoint;
- без базы данных;
- без OpenAI;
- без email-сервиса;
- без интернет-запросов.

## Требования к структуре

Используй текущую структуру проекта.

Предпочтительно разместить схемы и нормализаторы примерно так:

```text
app/
├── schemas/
│   ├── contact.py
│   └── normalizers.py
```

Допускается другое компактное размещение, если оно лучше соответствует текущему проекту.

Не создавай универсальные абстракции ради абстракций. Нужны конкретные функции с понятными именами:

```python
normalize_name()
normalize_email()
normalize_phone()
normalize_single_line_text()
normalize_multiline_text()
```

Функции можно объединить или изменить, если это уменьшит дублирование без ухудшения читаемости. Объясни решение в итоговом отчёте.

## Нормализация данных

Нормализация должна выполняться **до окончательной валидации, сохранения и дальнейшего использования данных**.

На этом этапе сохранения в БД ещё нет, но схема должна возвращать уже нормализованные значения.

### Имя

Для имени:

1. Удалить пробельные символы в начале и конце.
2. Последовательности пробельных символов внутри заменить одним обычным пробелом.
3. Нормализованное имя затем валидировать.
4. Вернуть нормализованное значение из схемы.

Пример:

```text
"   Иван    Иванов   " → "Иван Иванов"
```

### Email

Для email:

1. Удалить пробельные символы в начале и конце.
2. Привести значение к нижнему регистру.
3. Не пытаться исправлять внутренние пробелы.
4. Внутренние пробелы должны приводить к ошибке валидации.
5. Использовать стандартную надёжную валидацию email.

Пример:

```text
"  User@Example.COM  " → "user@example.com"
```

### Телефон

Для телефона:

1. Удалить пробельные символы в начале и конце.
2. Разрешить пользователю вводить телефон с:
   - пробелами;
   - дефисами;
   - круглыми скобками;
   - ведущим символом `+`.

3. Перед окончательной валидацией удалить декоративные символы.
4. Сохранять и возвращать телефон в едином нормализованном формате.
5. Российский номер, начинающийся с `8` и содержащий 11 цифр, преобразовывать в формат `+7`.
6. Российский номер, начинающийся с `7`, преобразовывать в формат `+7`.
7. Не ломать корректные международные номера.
8. Запрещать буквы и посторонние специальные символы.
9. Проверять разумное количество цифр.

Примеры:

```text
"8 (999) 123-45-67" → "+79991234567"
"+7 999 123 45 67" → "+79991234567"
"+49 30 123456" → "+4930123456"
```

Не требуется подключать тяжёлую библиотеку только ради телефона, если правила можно надёжно и понятно реализовать самостоятельно.

Если считаешь, что нужна внешняя библиотека, сначала оцени необходимость и объясни её в итоговом отчёте.

### Комментарий

Для комментария:

1. Удалить пробельные символы только в начале и конце всего значения.
2. Не схлопывать внутренние пробелы.
3. Не удалять переносы строк.
4. Не менять абзацы.
5. Не преобразовывать внутреннее форматирование.
6. Вернуть очищенное только по краям значение.

Пример:

Исходное значение:

```text
"   Первая строка.\n\nВторая    строка.   "
```

Результат:

```text
"Первая строка.\n\nВторая    строка."
```

### Однострочные поля

Для любых однострочных текстовых полей:

1. удалить пробелы по краям;
2. заменить последовательности пробельных символов одним пробелом;
3. использовать нормализованное значение для последующей валидации.

## Валидация имени

Имя должно быть реальным именем или сочетанием имени и фамилии, а не никнеймом.

Разрешить:

- русские буквы;
- латинские буквы;
- буквы других алфавитов, если стандартный Unicode-подход позволяет сделать это надёжно и понятно;
- одиночные пробелы между частями имени;
- дефисы внутри частей имени;
- апострофы внутри частей имени.

Примеры допустимых значений:

```text
Иван
Иван Иванов
Анна-Мария
Jean-Pierre
O'Connor
Мария де Ла Крус
```

Запретить:

```text
Иван123
12345
user_name
Admin!
Иван@
--Иван
Иван--
Иван  -  Петров
```

Дополнительные требования:

- длина после нормализации — от 2 до 80 символов;
- должна присутствовать хотя бы одна буква;
- имя не может начинаться или заканчиваться пробелом, дефисом или апострофом;
- разделители не должны идти подряд;
- цифры запрещены;
- посторонние специальные символы запрещены.

Не делай чрезмерно сложную валидацию, пытающуюся определить существование конкретного имени. Проверяется только безопасный и разумный формат.

Сообщения об ошибках должны быть конкретными и понятными на русском языке.

## Валидация телефона

Требования:

- обязательное поле;
- после нормализации содержит ведущий `+` и цифры;
- разумная длина международного номера;
- буквы запрещены;
- посторонние специальные символы запрещены;
- слишком короткий или слишком длинный номер отклоняется;
- сообщение ошибки должно объяснять причину.

Не привязывай приложение исключительно к российским номерам.

## Валидация email

Используй подходящую Pydantic-валидацию email.

Требования:

- обязательное поле;
- нормализация регистра;
- удаление пробелов только по краям;
- внутренние пробелы запрещены;
- длина должна быть разумно ограничена;
- сообщения об ошибках должны оставаться понятными клиенту API.

Если потребуется зависимость `email-validator`, добавь её в `requirements.txt`.

## Валидация комментария

Требования:

- обязательное поле;
- после удаления пробелов по краям не может быть пустым;
- минимальная длина — выбери разумное значение, например 5 символов;
- максимальная длина — выбери разумное значение, например 5000 символов;
- внутренние пробелы и переносы сохраняются;
- строка, состоящая только из пробелов и переносов, отклоняется;
- сообщение ошибки должно быть конкретным.

Зафиксируй выбранные ограничения в коде и `docs/test-plan.md`.

## Honeypot

Если добавляешь honeypot-поле:

- оно должно быть необязательным;
- обычный пользователь отправляет пустую строку или не передаёт его;
- заполненное поле должно считаться признаком спама;
- пока не реализуй HTTP-ответ и rate limiting;
- достаточно проверить это на уровне схемы или отдельного валидатора;
- не усложняй текущий этап ради honeypot.

Если считаешь, что honeypot логичнее добавить на этапе 7 вместе с формой и защитой от спама, не добавляй его сейчас и явно укажи это решение в отчёте.

Не создавай заранее большое количество схем для ещё не реализованных AI-результатов, email-статусов и модели БД.

Допустимо ограничиться входной схемой и вспомогательными типами текущего этапа.

В ключевых и неочевидных местах добавь комментарии на русском языке.

Комментарии должны объяснять:

Запрещено записывать в лог:

- полный телефон;

Ошибки валидации в дальнейшем будет логировать API-слой. Не создавай дублирующее и шумное логирование внутри каждого нормализатора.

- очередь;

## Типы писем

Создай два типа писем.

### 1. Уведомление владельцу

Письмо должно содержать:

- имя пользователя;
- email;
- телефон;
- комментарий;
- дату и время обращения, если они переданы;
- категорию обращения;
- тональность;
- приоритет;
- краткое AI-резюме;
- предлагаемый AI-ответ;
- понятную отметку, если использован AI fallback.

Не все AI-поля обязательно присутствуют. Шаблон должен корректно работать, если:

- summary отсутствует;
- категория неизвестна;
- AI fallback применён;
- AI-анализ ещё не выполнялся.

Не выводи `None` в письме.

### 2. Подтверждение пользователю

Письмо должно:

- обратиться к пользователю по имени;
- подтвердить получение обращения;
- поблагодарить за сообщение;
- не обещать конкретный срок ответа;
- не обещать выполнение работы;
- не раскрывать внутреннюю классификацию;
- не раскрывать технические ошибки AI или email;
- использовать AI `suggested_reply`, если он доступен и валиден;
- использовать безопасный шаблонный ответ при отсутствии AI-ответа.

Безопасный fallback-текст:

```text
Спасибо за обращение. Сообщение получено, я ознакомлюсь с ним и свяжусь с вами при первой возможности.
```

Текст можно немного улучшить, но без конкретных обещаний.

## Порядок pipeline

Реализуй следующий порядок.

### 1. Создание обращения

После валидации:

- сохранить нормализованные данные;
- получить `contact_id`;
- начальные статусы берутся из модели;
- записать понятный лог.

Если сохранить обращение не удалось:

- выполнить rollback на уровне repository;
- вернуть безопасную серверную ошибку;
- не вызывать AI;
- не отправлять письма.

### 2. Начало обработки

После создания:

```text
processing_status = processing
```

Если обновление статуса неожиданно не удалось, обработай это как критическую внутреннюю ошибку, потому что основная БД уже работает нестабильно.

Не запускай внешние сервисы при подтверждённой критической ошибке БД.

### 3. AI-анализ

Передай в AI-сервис только комментарий.

AI-сервис возвращает типизированный результат.

При успехе:

- сохранить AI-поля;
- сохранить `ai_status=success`.

При fallback:

- сохранить безопасные fallback-поля;
- сохранить `ai_status=fallback`;
- сохранить безопасный внутренний `ai_error`, если модель это поддерживает.

`ai_error` не должен содержать:

- API-ключ;
- полный traceback;
- комментарий;
- персональные данные;
- чрезмерно длинный ответ провайдера.

Если fake AI-сервис выбрасывает исключение вместо возврата результата, `ContactService` обязан перехватить исключение и самостоятельно применить fallback.

То есть orchestration не должен полностью доверять тому, что нижний сервис всегда возвращает корректный объект.

### 4. Формирование email-контекста

После AI-этапа сформируй единый типизированный email-контекст из:

- сохранённого обращения;
- результата AI;
- текущих статусов.

Не передавай ORM-модель напрямую в Jinja2, если это ухудшает контроль данных.

Предпочтительно использовать существующую Pydantic-схему email-контекста.

### 5. Письмо владельцу

Сначала сформируй и отправь письмо владельцу.

Результат независимо сохранить в:

- `owner_email_status`;
- `owner_email_error`.

Если email-сервис выбрасывает неожиданное исключение:

- перехватить;
- сохранить `failed`;
- сохранить безопасный error code/message;
- продолжить pipeline;
- перейти к письму пользователю.

### 6. Письмо пользователю

Отправить независимо от результата письма владельцу.

Сохранить:

- `user_email_status`;
- `user_email_error`.

Ошибка письма пользователю не должна отменять письмо владельцу и наоборот.

### 7. Итоговый processing status

Определи итоговый статус по понятным правилам.

Рекомендуемая политика:

```text
completed
```

если:

- AI success;
- письмо владельцу sent;
- письмо пользователю sent.

```text
completed_with_errors
```

если обращение сохранено, но:

- AI fallback;
- хотя бы одно письмо failed;
- хотя бы одно письмо skipped;
- произошла некритическая ошибка внешнего компонента.

```text
failed
```

только если:

- основная внутренняя обработка после сохранения не может быть завершена из-за критической ошибки БД;
- сохранить итоговые состояния невозможно.

Не возвращай `failed` только потому, что OpenAI или Resend недоступен.

### 8. Ответ

Верни HTTP 201 и безопасную response schema.

Сообщение пользователю:

```text
Обращение принято
```

Можно использовать чуть более дружелюбный текст, но он должен оставаться нейтральным.

## Частичные ошибки БД

Раздели ошибки внешних сервисов и ошибки сохранения статусов.

Если OpenAI недоступен, но fallback сохранён — pipeline продолжается.

### Правила частичных ошибок

AI fallback, AI exception и email failed/skipped не теряют обращение и не превращают успешное сохранение в HTTP 500. Эти состояния сохраняются и дают `completed_with_errors`.

### Правила итогового processing status

- `completed`: AI success, owner email sent, user email sent.

## Определение IP-адреса

На Render приложение будет находиться за reverse proxy.

Нужно корректно определять адрес клиента.

Предпочтительный порядок:

1. использовать доверенный proxy header только при `TRUST_PROXY_HEADERS=true`;
2. рассматривать `X-Forwarded-For`;
3. брать первый валидный адрес в цепочке;
4. при отсутствии заголовка использовать `request.client.host`;
5. при невозможности определить адрес использовать безопасное значение, но явно логировать проблему.

Не доверяй произвольному `X-Forwarded-For`, если поддержка proxy headers отключена.

Не логируй полный IP без необходимости, если это противоречит текущей политике проекта. Допустимо:

* хранить IP внутри limiter;
* логировать маскированное значение;
* логировать стабильный короткий hash IP.

Предпочтительный вариант для логов — короткий hash:

```text
client_key=ip_sha256:8f23ab91
```

Это позволяет связывать события без раскрытия полного адреса.

Добавь русский комментарий, объясняющий доверие proxy headers.

## Rate limiter

Limiter должен проверяться **до вызова `ContactService`**.

Для каждого IP разрешено, например:

```text
3 запроса за 600 секунд
```

После превышения:

* вернуть HTTP 429;
* не вызывать `ContactService`;
* не создавать запись;
* не вызывать AI;
* не отправлять email;
* вернуть безопасный ответ в общем формате ошибок;
* вернуть `request_id`;
* по возможности добавить `Retry-After`.

Пример body:

```json
{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Слишком много обращений. Попробуйте повторить позже.",
    "details": []
  },
  "request_id": "..."
}
```

Не сообщай клиенту:

* внутренний ключ limiter;
* точный IP;
* содержимое хранилища;
* traceback.

## Обработка ошибок

Ошибка самого limiter не должна молча пропускать спам.

Выбери и объясни политику:

* fail closed — временно отклонить запрос;
* fail open — пропустить запрос и залогировать ошибку.

Для локального in-memory limiter его ошибки означают внутреннюю проблему. Предпочтительно:

* залогировать traceback;
* вернуть безопасный HTTP 500;
* не вызывать pipeline.

Не создавай абстрактное сообщение `Rate limit error`.

Лог должен содержать:

* этап;
* тип исключения;
* request ID;
* факт, что pipeline не вызван.

### Выбранная реализация

- Собственный компактный in-memory limiter.
- Алгоритм: sliding window.

### Определение client IP

- При `TRUST_PROXY_HEADERS=true` используется `X-Forwarded-For`.
- Из цепочки берётся первый валидный IP.
- При `TRUST_PROXY_HEADERS=false` header игнорируется.
- При отсутствии валидного proxy header используется `request.client.host`.
- При невозможности определить адрес используется `unknown-client` и логируется `client_ip_resolution_failed`.

### Маскирование IP

- Полный IP используется только внутри limiter.
- В логах используется короткий SHA-256 hash: `client_key=ip_sha256:xxxxxxxx`.
- Полный IP, пользовательские поля и honeypot value не логируются.

### HTTP 429

- Ответ возвращается до `ContactService`.
- Body:

```json
{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Слишком много обращений. Попробуйте повторить позже.",
    "details": []
  },
  "request_id": "..."
}
```

- Header `Retry-After` добавляется, если рассчитан.

### Honeypot

- Поведение схемы сохранено: заполненный `website` возвращает HTTP 422.
- Pipeline не вызывается.
- Добавлено безопасное событие `event=contact_honeypot_triggered`.

### Структура metrics

- `processing`
- `ai`
- `emails.owner`
- `emails.user`
- `categories`
- `generated_at`
- `request_id`

Все известные enum-статусы и категории присутствуют в ответе даже при нулевых значениях. Старые или неизвестные значения сводятся в `unknown` и логируются.

### Диагностические статусы health

- `ok`: БД доступна, AI/email включены и локально настроены.
- `degraded`: БД доступна, но AI/email отключены или не настроены.
- `unavailable`: БД недоступна, HTTP 503.

AI/email проверяются только по локальной конфигурации. Внешние запросы к ProxyAPI и Resend не выполняются.

### Изменения repository

- `ContactRepository.get_metrics()` остаётся единственным источником агрегатов.
- Используются SQL `count/group_by`, без загрузки всех обращений в Python.
- Пустые/старые категории не отбрасываются, чтобы API мог показать их как `unknown`.
- При SQLAlchemy-ошибке выполняется rollback и поднимается `ContactRepositoryError`.
