# Правила разработки DevReach AI

## Общий порядок работы

- Перед изменениями изучать существующий проект.
- Перед каждой итерацией читать `project-rules.md`.
- Выполнять только текущий согласованный этап.
- Не переписывать рабочую архитектуру без необходимости.
- Не добавлять лишние функции вне текущего этапа.
- Не выполнять `git commit`.

## Python и инфраструктура

- Использовать Python 3.12.
- Локальный запуск выполнять через `.venv`.
- Docker и Docker Compose не используются.
- Деплой будет выполняться на Render вручную через Web Service.
- `render.yaml` не используется.

## Код

- Ключевые и неочевидные места комментировать по-русски.
- Комментарии должны объяснять причины решений, а не пересказывать очевидные строки.
- Не создавать бессмысленные абстракции.
- Использовать конкретные имена функций, классов, событий и ошибок.

## Нормализация

- Все однострочные текстовые поля обрезаются по краям.
- Повторяющиеся пробельные символы в однострочных полях заменяются одним пробелом.
- Нормализация выполняется до валидации, сохранения и использования.
- В многострочном комментарии удаляются только пробельные символы в начале и конце.
- Внутренние пробелы, переносы и абзацы комментария сохраняются.

## Логирование

- Логировать начало, успех и ошибку важных операций.
- Писать конкретную причину ошибки.
- Использовать `request_id` и `contact_id`, когда `contact_id` появится.
- Логировать применение fallback.
- Не логировать секреты и полный пользовательский комментарий.

## Внешние сервисы

- Реальные запросы OpenAI и реальные письма запрещены по умолчанию.
- Live-вызовы разрешены только отдельной командой с явным флагом.
- Перед live-командой явно предупреждать о токенах или реальном письме.

## Тестирование

- Каждый тест должен иметь понятное имя.
- Каждый тест должен иметь краткое русское описание.
- Все тесты должны иметь понятные `ids` на русском языке. Вывод в консоль должен быть примерно такого вида
  test_debug_value_is_parsed[debug включается значением true] PASSED
- Вывод pytest должен позволять понять, что проверяется.
- Каждый тест добавляется в `docs/test-plan.md`.
- После этапа запускаются только релевантные тесты.
- Полный прогон выполняется только на контрольных точках.

## Документация

- README полноценно создаётся на финальном этапе.
- `.env.example` обновляется при добавлении переменных.
- `.env` не попадает в git.
- Каждый промпт фиксируется в AI-log.
- После этапа Codex обязан выдать точные команды проверки.

## Формат итогового отчёта Codex

1. Что реализовано.
2. Изменённые файлы.
3. Архитектурные решения.
4. Основная команда проверки.
5. Дополнительная ручная команда проверки.
6. Ожидаемый результат.
7. Что не запускалось.
8. Были ли внешние вызовы.
9. Результат тестов.
10. `git status`.
11. Известные ограничения.
12. Подтверждение остановки после текущего этапа.
